# Tasks 디렉토리 구조

Yesman-Claude 프로젝트의 작업 관리를 위한 체계적인 디렉토리 구조입니다.

## 📁 디렉토리 구조

```
tasks/
├── README.md          # 이 파일 - 구조 설명
├── done/               # ✅ 완료된 작업들
├── todo/               # 📋 현재 진행해야 할 작업들  
└── backlog/            # 🔮 미래에 계획된 작업들
```

## 📋 작업 분류 기준

### ✅ done/ - 완료된 작업
- 이미 구현되고 테스트가 완료된 기능들
- 과거 개발 이력 및 성과 기록
- 참고 자료 및 학습 목적

### 📋 todo/ - 현재 작업 대상
- **우선순위 HIGH**: 즉시 진행해야 할 핵심 기능
- **구체적 실행 계획**: 상세한 구현 방법 및 시간 추정
- **명확한 성공 기준**: 완료 여부를 판단할 수 있는 기준

### 🔮 backlog/ - 미래 계획
- **장기 비전**: 프로젝트의 미래 방향성
- **혁신적 아이디어**: 실험적이고 도전적인 기능들
- **요구사항 정의 필요**: 구체화가 더 필요한 개념들

### alert/ - 부적합 작업

## 📝 작업 파일 작성 규칙

### 파일 구조
각 작업 파일은 다음 구조를 따릅니다:

```markdown
# 작업 제목

## 작업 개요
**상태**: 📋 대기중 / ✅ 완료 / 🔮 미래 계획
**우선순위**: HIGH / MEDIUM / LOW
**예상 소요시간**: X시간
**복잡도**: HIGH / MEDIUM / LOW

## 목표
작업의 명확한 목표와 기대 효과

## 상세 요구사항
구체적인 기능 요구사항들

## 구현 계획
단계별 구현 방법

## 성공 기준
완료 여부를 판단할 수 있는 기준들

## 의존성
선행 작업이나 관련 작업들

---
추가 설명이나 참고사항
```

### 파일명 규칙
- **kebab-case 사용**: `feature-name-implementation.md`
- **의미있는 이름**: 작업 내용을 쉽게 파악할 수 있는 이름
- **일관된 접미사**: 작업 유형에 따른 접미사 사용

## 🔄 작업 플로우

### 새 작업 추가
1. `todo/` 디렉토리에 새 파일 생성
2. 표준 템플릿에 따라 내용 작성
3. 우선순위 및 의존성 설정

### 작업 진행
1. `todo/` 파일의 상태를 "진행중"으로 업데이트
2. 구현 과정의 주요 변경사항 기록
3. 문제 발생 시 해결 방법 문서화

### 작업 완료
1. 성공 기준 달성 확인
2. `done/` 디렉토리로 파일 이동
3. 완료일 및 성과 기록 추가

### 작업 연기
1. 연기 사유 명확히 기록
2. `backlog/`으로 이동 또는 우선순위 조정
3. 재개 조건 명시

## 🎯 관리 원칙

1. **하나의 파일, 하나의 작업**: 각 파일은 명확히 구분되는 단일 작업만 다룸
2. **구체적 실행 계획**: 모호한 목표보다는 실행 가능한 계획 중심
3. **지속적 업데이트**: 작업 진행에 따른 실시간 상태 반영
4. **학습 기록**: 완료된 작업의 경험과 교훈 보존
